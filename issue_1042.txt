### Anything else?

@MaizeShark reported this issue over in my fork of the AutoKey repository but I believe it belongs here instead.  This is an issue I've thought about myself.  AutoKey for Wayland currently only supports the Gnome desktop.  At some point it needs to be extended to the other desktop environments, like KDE.

@sebastiansam55 did refactor the AutoKey code to make this a bit easier.  The desktop-window-specific functions of AutoKey have been broken out into their own modules.  There's an abstract class definition that serves as a template for these and instances of that class exist for X11 and Gnome.  Supporting KDE, or anything else, only needs a new class to be created that interacts with the facilities of that specific desktop environment.  

For future reference, here's where the code resides today:

The AbstractWindowInterface class -> [lib/autokey/sys_interface/abstract_interface.py](https://github.com/dlk3/autokey/blob/main/lib/autokey/sys_interface/abstract_interface.py)
The X11-specific instance of that abstract class -> [lib/autokey/interface.py](https://github.com/dlk3/autokey/blob/main/lib/autokey/interface.py)
The Gnome-specific instance of that abstract class -> [lib/autokey/gnome_interface.py](https://github.com/dlk3/autokey/blob/main/lib/autokey/gnome_interface.py)
The code that determines which desktop we're running under and which class gets used -> [lib/autokey/iomediator/iomediator.py](https://github.com/dlk3/autokey/blob/main/lib/autokey/iomediator/iomediator.py)
